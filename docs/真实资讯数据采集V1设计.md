# PulseBrief 真实资讯数据采集 V1 设计

## 背景与现状

PulseBrief 当前已经完成 Flutter 用户端高保真 UI、Spring Boot V1 用户端 API、MySQL/Flyway、用户中心、历史列表、分页规范和 OpenAPI/Swagger。现有用户端内容主要来自 Flyway 种子数据，适合验证页面和接口，但还不能支撑真实资讯产品的日常运营。

真实资讯产品需要解决两个核心问题：

1. 资讯从哪里来。
2. 采集后的内容如何被审核、编辑并发布到用户端。

因此下一阶段不应只做 React Admin，也不应直接做无审核的自动采集发布，而应先搭建“真实资讯采集 + 候选池 + Admin 审核发布”的基础闭环。

## 目标与非目标

目标：

1. 设计真实资讯采集 V1 的数据源边界、数据模型和处理流程。
2. 设计原始资讯池、候选资讯池、采集任务日志和去重机制。
3. 明确 V1 只采集公开可引用信息，不展示新闻全文。
4. 明确所有采集内容必须经 Admin 审核后才能发布到用户端。
5. 预留后续接入多个数据源 Provider 的扩展点。
6. 为后续 Spring Boot 实现和 React Admin 审核页面提供边界。

非目标：

1. 不在 V1 抓取付费研报、PDF、完整新闻正文或需要登录授权的内容。
2. 不绕过目标网站反爬策略。
3. 不实现自动发布到用户端。
4. 不实现生产级 AI 摘要服务。
5. 不实现复杂推荐算法。
6. 不实现多管理员 RBAC。
7. 不在 V1 接入真实支付、会员或研报下载。

## 影响范围

后端：

1. 新增采集源配置、原始资讯、采集任务日志、候选资讯相关表。
2. 新增采集服务、去重服务、候选生成服务。
3. 后续新增 Admin API，用于触发采集、查看候选、审核发布。
4. OpenAPI 后续需要增加 Admin 分组。

Flutter：

1. Flutter 用户端短期不直接感知采集系统。
2. 用户端仍只读取已发布 `news_article` 和 `daily_digest`。
3. 当 Admin 发布新文章后，现有首页、分类页、详情页可读取真实发布内容。

Admin：

1. 后续需要候选资讯列表。
2. 后续需要候选详情、编辑、拒绝、发布。
3. 后续需要采集任务日志和失败原因展示。

部署与配置：

1. 新增外部数据源 API Key 或 RSS URL 配置。
2. API Key 只能通过环境变量或本地未提交配置注入。
3. 采集任务默认可关闭，避免开发环境频繁请求外部服务。

## 数据模型与权限模型

### 数据模型

#### news_ingestion_source

采集源配置表。

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `code` | 采集源编码，如 `gdelt_global`、`rss_ai_official` |
| `name` | 采集源名称 |
| `provider_type` | Provider 类型，如 `API`、`RSS` |
| `base_url` | API 或 RSS 基础地址 |
| `default_category_code` | 默认分类 |
| `enabled` | 是否启用 |
| `rate_limit_per_hour` | 每小时请求上限 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

#### news_ingestion_job

采集任务日志表。

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `source_code` | 采集源编码 |
| `trigger_type` | `MANUAL` 或 `SCHEDULED` |
| `status` | `RUNNING`、`SUCCESS`、`PARTIAL_SUCCESS`、`FAILED` |
| `started_at` | 开始时间 |
| `finished_at` | 结束时间 |
| `fetched_count` | 拉取数量 |
| `new_count` | 新增数量 |
| `duplicate_count` | 重复数量 |
| `candidate_count` | 生成候选数量 |
| `error_message` | 错误信息 |

#### raw_news_item

原始资讯池。

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `source_code` | 采集源编码 |
| `provider_item_id` | 外部数据源返回的条目 ID，可为空 |
| `title` | 标题 |
| `summary` | 外部源摘要或描述 |
| `source_name` | 原始来源名称 |
| `original_url` | 原文链接 |
| `original_url_hash` | 原文链接去重哈希 |
| `image_url` | 原始图片链接，可为空 |
| `published_at` | 外部发布时间 |
| `fetched_at` | 采集时间 |
| `language` | 语言 |
| `country` | 国家或地区 |
| `raw_payload` | 原始 JSON，V1 只存必要元数据 |
| `content_hash` | 去重指纹 |
| `status` | `NEW`、`DUPLICATE`、`CANDIDATE`、`REJECTED`、`PUBLISHED`、`ERROR` |
| `duplicate_of_id` | 重复目标 ID |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

#### candidate_article

候选资讯池。

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `raw_news_item_id` | 原始资讯 ID |
| `title` | 候选标题，可被 Admin 编辑 |
| `summary` | 候选摘要，可被 Admin 编辑 |
| `category_code` | 分类编码 |
| `source_name` | 来源名称 |
| `original_url` | 原文链接 |
| `published_at` | 原文发布时间 |
| `status` | `PENDING_REVIEW`、`REJECTED`、`PUBLISHED` |
| `review_note` | 审核备注 |
| `published_article_id` | 发布后的文章 ID |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 权限模型

1. 用户端接口仍使用现有开发态 Bearer Token。
2. 采集触发、候选审核、发布、拒绝都属于 Admin 写操作。
3. V1 Admin 可先使用开发态 Admin Token，后续再替换为正式后台登录。
4. 用户端不能访问 `raw_news_item` 和 `candidate_article`。

## 后端实现方案

### 模块划分

建议新增包：

```text
com.pulsebrief.ingestion
├── api
├── domain
├── repository
├── service
└── provider
```

职责：

1. `provider`：对接外部数据源，返回统一 `RawNewsPayload`。
2. `service`：负责采集流程、去重、候选生成。
3. `repository`：持久化采集源、任务日志、原始资讯、候选资讯。
4. `api`：先预留给 Admin API，不开放给用户端。

### Provider 抽象

```java
public interface NewsIngestionProvider {
    String providerType();

    List<RawNewsPayload> fetch(NewsIngestionSource source, IngestionRequest request);
}
```

`RawNewsPayload` 字段：

```text
providerItemId
title
summary
sourceName
originalUrl
imageUrl
publishedAt
language
country
rawPayload
```

### 去重策略

V1 分三层：

1. 原文链接完全相同：直接判定重复。
2. `sourceName + title normalized + publishedDate` 相同：判定重复。
3. 标题指纹相似但链接不同：V1 先不自动合并，进入候选池时标记可能重复，交由 Admin 判断。

标题 normalized 规则：

1. 转小写。
2. 去除首尾空格。
3. 压缩连续空白。
4. 去除常见标点。

### 状态流转

```text
RAW NEW
→ DUPLICATE
→ CANDIDATE
→ PENDING_REVIEW
→ PUBLISHED

PENDING_REVIEW
→ REJECTED
```

说明：

1. `raw_news_item` 负责记录采集事实。
2. `candidate_article` 负责审核工作流。
3. `news_article` 负责用户端正式发布内容。

## 测试与回归方案

后端测试：

1. Provider fixture 测试：固定外部 JSON 映射为 `RawNewsPayload`。
2. 去重测试：相同 URL 不重复入库。
3. 去重测试：相同来源、标题和日期不重复入库。
4. 采集任务日志测试：成功、部分成功、失败都写入日志。
5. 候选生成测试：新原始资讯可生成候选。
6. 发布隔离测试：未审核候选不会出现在用户端文章 API。

Flutter 测试：

1. 本阶段不要求新增 Flutter 测试。
2. 后续发布闭环完成后，继续使用 live API 测试验证新发布文章可读取。

Admin 测试：

2. 候选详情可打开。

## 风险与分阶段落地建议

### 风险

1. 外部 API 额度、限流和稳定性不可控。
2. 新闻版权边界需要持续控制。
3. 自动摘要可能产生事实错误，V1 不自动发布。
4. 多来源重复内容可能较多，需要逐步优化去重。
5. 采集内容质量参差不齐，需要 Admin 审核兜底。

### 分阶段落地

第一轮：

1. 采集适配层与 fixture Provider。已完成。
2. 建表与采集任务日志。已完成。
3. 原始资讯入库。已完成。
4. URL 去重和标题指纹去重。已完成。
5. 候选资讯生成。

第二轮：

1. Admin API。
2. 候选资讯审核 UI。
3. 发布为 `news_article`。
4. Flutter 用户端读取发布结果。

第三轮：

1. 增加更多数据源。
2. 增加分类规则。
3. 增加每日简报编辑。
4. 增加操作日志和基础监控。

## 第一批建议数据源

V1 第一批只选一个低风险来源：

1. 已先落地 `FIXTURE` Provider，验证抽象层和测试链路。
2. 下一批优先官方 RSS。
3. 再评估公开新闻 API。
4. 所有真实来源都必须不需要登录、不抓全文、只返回标题摘要和链接。

财经、投行观点和 AI 动态后续分批接入。投行观点必须只采集公开网页或官方公开摘要，不采集付费研报和 PDF。
