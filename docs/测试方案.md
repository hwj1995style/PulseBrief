# PulseBrief 测试方案

## 测试目标

验证正式 V1 工程主线是否保持可构建、可测试，并为后续 Spring Boot、Flutter、MySQL 联调保留稳定回归基线。

## 自动化测试

### 后端测试

覆盖范围：

1. Spring Boot 应用入口存在且可调用。
2. `/api/health` 返回服务状态。
3. 分类、文章、简报、收藏、订阅、播放历史测试按 V1 接口分批补充。

V1 核心接口测试：

1. 分类：`GET /api/categories` 返回启用分类并按 `sortNo` 排序。
2. 首页：`GET /api/articles/home` 返回今日简报、投行观点推荐和资讯列表。
3. 文章：`GET /api/articles` 支持分类分页；`GET /api/articles/{id}` 返回结构化摘要且不返回新闻全文。
4. 简报：`GET /api/digests/today` 和 `GET /api/digests/{id}` 返回播报所需字段。
5. 登录：`POST /api/auth/login` 校验协议同意状态；`POST /api/auth/guest` 支持游客模式。
6. 订阅：未登录返回 401；登录后可查询和保存订阅。
7. 收藏：收藏与取消收藏保持幂等。
8. 播放历史：文章和简报播放记录均可写入。
9. Admin 发布候选后，用户端 `/api/articles` 和 `/api/articles/{id}` 能读取发布文章。
10. Admin 发布每日简报后，用户端 `/api/digests/today` 和 `/api/digests/{id}` 能读取发布简报。
11. Admin 采集监控接口可查询采集任务、失败错误信息、今日统计和采集源状态，并可切换采集源启停。
12. Admin 发布候选文章、发布每日简报和下线每日简报后会写入发布操作日志，且 `/api/admin/operation-logs` 可查询。
13. Admin 异常数据检测接口可返回原始资讯缺来源、缺原文链接、缺发布时间和未来发布时间异常。

数据库回归：

1. Flyway migration 在空库可初始化。
2. 种子数据包含分类、来源、文章和简报。
3. `news_article.article_hash` 唯一约束生效。

### Flutter 测试

覆盖范围：

1. 当前默认 widget 测试保持通过。
2. 后续补充首页、分类、详情、每日简报、收藏、播放器入口的 widget 测试。
3. UI 稳定后补充 golden 测试。

### Flutter Repository 数据层测试

覆盖范围：

1. 默认数据源为 mock，确保本机预览不依赖后端。
2. `PULSEBRIEF_DATA_SOURCE=api` 时创建 API repository。
3. API repository 可将 Spring Boot V1 响应映射为 Flutter UI model。
4. 登录、游客模式、分类、首页、简报、订阅快照均有最小单元测试。
5. 后台发布文章详情响应可映射为 APP `Article` 模型，包括 AI 摘要、核心要点和影响分析。
6. 后台发布简报响应可映射为 APP `TodayDigestFeed` 和 `Digest` 模型，包括热点清单和播报文案。

## CI 验证

CI 设计见 `docs/CI验证流水线设计.md`。

GitHub Actions 在 `main` 分支 push 和 pull request 时运行：

1. `backend` job：启动 MySQL 8.4 service container，执行 `backend/` 下的 `./mvnw test`。
2. `admin` job：执行 `npm ci`、`npm test -- --run`、`npm run lint` 和 `npm run build`。
3. `mobile` job：执行 `flutter pub get`、`flutter analyze` 和 `flutter test`。
4. `deploy-config` job：执行 `docker compose -f deploy/docker-compose.yml config`。

CI 默认不运行 Flutter live API 测试、不构建 Android APK、不接真实 Provider 或 AI 服务。需要真实密钥或外部网络的检查必须单独设计环境变量和跳过策略，不得直接写入 workflow。

## 真实 Provider 配置检查

真实 RSS/API Provider 接入前必须先通过本地配置检查，不得直接提交密钥或让 CI 请求外部数据源。

默认安全配置检查：

```powershell
.\scripts\check-provider-env.ps1
.\scripts\check-provider-env.ps1 -EnvFile .\.env.example
```

负向配置检查：

1. 启用 RSS 但使用 `example.com` 占位 URL，应失败。
2. 启用 API 但缺少 `PULSEBRIEF_PROVIDER_API_KEY`，应失败。
3. `PULSEBRIEF_INGESTION_ALLOW_BACKFILL=true` 应失败。
4. 全文或 PDF 能力开启但许可策略不匹配，应失败。

## 真实 RSS Provider V1 测试

阶段 23 自动化测试只使用本地 RSS/Atom fixture，不访问真实外网：

```powershell
cd backend
.\mvnw.cmd "-Dtest=RssNewsIngestionProviderTest,NewsIngestionProviderContextTest,RssNewsIngestionProviderLiveTest" test
```

预期结果：

1. RSS/Atom fixture 解析测试通过。
2. Spring 上下文同时注册 `FIXTURE` 和 `RSS` Provider。
3. `RssNewsIngestionProviderLiveTest` 默认跳过，不请求真实外网。

真实外网采集验证只作为本地手动 live smoke，必须显式开启：

```powershell
$env:PULSEBRIEF_RSS_LIVE_TEST_ENABLED='true'
$env:PULSEBRIEF_RSS_LIVE_TEST_URL='<public-rss-url>'
cd backend
.\mvnw.cmd "-Dtest=RssNewsIngestionProviderLiveTest" test
Remove-Item Env:PULSEBRIEF_RSS_LIVE_TEST_ENABLED
Remove-Item Env:PULSEBRIEF_RSS_LIVE_TEST_URL
```

阶段 23 本机验证记录：

1. 验证时间：2026-06-15。
2. 验证 URL：`https://rss.nytimes.com/services/xml/rss/nyt/World.xml`。
3. 验证命令：开启 `PULSEBRIEF_RSS_LIVE_TEST_ENABLED=true` 后执行 `RssNewsIngestionProviderLiveTest`。
4. 验证结果：Java RSS Provider 成功拉取真实外网 RSS，解析 1 到 5 条元数据并通过断言。
5. 网络说明：本机外网访问依赖 `HTTP_PROXY` / `HTTPS_PROXY`，`HttpRssFeedClient` 已读取这些环境变量；CI 仍不运行 live smoke。

live smoke 不进入 CI，不提交 `.env.local`、采集输出文件、PDF 或数据库 dump。
