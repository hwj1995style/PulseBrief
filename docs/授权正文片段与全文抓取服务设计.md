# 授权正文片段与全文抓取服务设计

## 背景与现状

PulseBrief 已完成真实 RSS Provider V1 和 Admin 手动触发真实 Provider 闭环。当前链路可以从数据库配置的 RSS source 拉取真实外网 RSS 元数据，写入 `raw_news_item`，生成 `candidate_article`，并在 Admin 中审核发布。

阶段 23 和阶段 24 已明确 RSS Provider 只负责元数据采集，不抓取网页全文、不下载 PDF、不调用 AI 摘要、不做复杂分类。现在进入阶段 25，需要为后续更完整内容能力设计独立的正文抓取服务，让系统在不破坏 Provider 边界、不扩大合规风险的前提下，支持授权正文片段或授权全文进入候选审核流程。

已有基础：

1. `news_ingestion_source` 已有 `content_access_policy`、`max_age_hours`、`allow_full_text` 和 `license_note`。
2. `RawNewsIngestionService` 已按采集源最新窗口过滤过期内容。
3. `candidate_article` 已作为 Admin 审核发布入口。
4. `news_ingestion_job` 已能记录手动采集任务成功和失败。
5. `docs/真实RSSProviderV1设计.md` 已把 `ContentFetchService` 列为后续独立阶段。

## 目标与非目标

目标：

1. 设计独立的 `ContentFetchService`，只处理已经入库的 `raw_news_item` 或候选内容。
2. 明确正文抓取只允许 `SNIPPET_ALLOWED` 或 `FULLTEXT_ALLOWED` 且具备授权说明的 source。
3. 设计正文片段、授权全文、正文哈希、抓取状态和错误信息的数据模型扩展。
4. 设计正文抓取任务生命周期、限流、超时、失败降级和可观测日志。
5. 设计 Admin 合规审核入口：可查看来源许可、正文预览、抓取错误和发布边界。
6. 设计测试策略：自动化测试只使用本地 HTML fixture，真实网页抓取只允许本地显式 smoke。

非目标：

1. 不抓取未授权新闻全文。
2. 不绕过登录、付费墙、验证码、Cookie、反爬、robots 或访问控制。
3. 不批量回溯历史网页。
4. 不下载、缓存或解析 PDF；PDF 放到阶段 26。
5. 不接入 AI 摘要；AI 摘要放到阶段 27。
6. 不做复杂正文抽取算法平台；V1 只做有限、可控、可测试的正文片段/授权全文抓取。
7. 不让用户端默认展示全文；是否展示授权全文由后续发布策略决定。

## 影响范围

后端：

1. 新增正文抓取服务和 HTML 解析适配层。
2. 扩展原始资讯或新增正文内容表，用于保存正文片段/授权全文及抓取状态。
3. 新增正文抓取任务日志或扩展现有 job 记录，保证失败可定位。
4. 候选详情接口可返回正文预览、许可状态和抓取状态。
5. 发布接口需要校验合规状态，未授权正文不能进入用户端文章。

Admin：

1. 候选详情页展示正文许可策略、授权说明、正文预览、抓取状态和错误信息。
2. 支持 Admin 选择“仅保留原文链接”“保留正文片段”“确认授权全文”。
3. 对未授权 source，正文相关区域只展示禁止原因，不提供抓取动作。

Flutter：

1. 阶段 25 设计不直接修改用户端。
2. 后续若允许展示授权全文，详情页应默认仍以 AI 摘要和结构化要点为主，授权全文放入折叠区或原文入口之后。

配置与部署：

1. 默认不启用真实网页抓取。
2. 真实网页 smoke 需要显式环境开关。
3. CI 不访问真实网页、不依赖代理、不依赖第三方解析服务。

## 数据模型或权限模型

### 来源许可

正文抓取前必须同时满足：

1. `news_ingestion_source.enabled = true`。
2. `content_access_policy in ('SNIPPET_ALLOWED', 'FULLTEXT_ALLOWED')`。
3. `license_note` 非空，且能说明授权来源。
4. 原始条目在 source 的 `max_age_hours` 最新窗口内。
5. `original_url` 是公开可访问 URL，且不需要登录或绕过访问控制。

策略含义：

| 策略 | 阶段 25 行为 |
| --- | --- |
| `SUMMARY_ONLY` | 不抓正文，只保留 RSS/API 摘要和原文链接 |
| `SNIPPET_ALLOWED` | 可抓取有限正文片段，默认上限 500 到 1000 字 |
| `FULLTEXT_ALLOWED` | 可保存授权全文，但仍需 Admin 审核后发布 |
| `PDF_ALLOWED` | 不在本阶段处理，交给阶段 26 |
| `LINK_ONLY` | 不抓正文，只保留标题和链接 |
| `UNKNOWN` | 按 `LINK_ONLY` 处理 |

### 数据模型建议

方案 A：扩展 `raw_news_item`

| 字段 | 说明 |
| --- | --- |
| `content_capture_mode` | `NONE/SNIPPET/FULLTEXT` |
| `content_text` | 正文片段或授权全文 |
| `content_text_hash` | 正文内容哈希，用于去重和变更检测 |
| `content_license_policy` | 抓取时使用的许可策略快照 |
| `content_license_note` | 抓取时的授权说明快照 |
| `content_fetch_status` | `NOT_REQUIRED/PENDING/SUCCESS/FAILED/SKIPPED` |
| `content_fetch_error` | 抓取或解析失败原因 |
| `content_fetched_at` | 正文抓取时间 |

优点：查询简单，候选生成可直接读取。

缺点：`raw_news_item` 会继续变宽，后续多版本正文、正文解析结果和 AI 输入审计不够灵活。

方案 B：新增 `raw_news_content` 表

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `raw_news_item_id` | 原始资讯 ID |
| `source_code` | 采集源编码 |
| `capture_mode` | `SNIPPET/FULLTEXT` |
| `content_text` | 正文片段或授权全文 |
| `content_text_hash` | 正文内容哈希 |
| `license_policy` | 许可策略快照 |
| `license_note` | 授权说明快照 |
| `fetch_status` | `PENDING/SUCCESS/FAILED/SKIPPED` |
| `error_message` | 错误信息 |
| `fetched_at` | 抓取时间 |
| `created_at/updated_at` | 审计时间 |

优点：边界清晰，便于后续 AI 摘要、正文版本、解析审计和全文清理。

缺点：查询和候选详情映射稍复杂。

推荐采用方案 B：新增 `raw_news_content`。阶段 25 是内容增强阶段，正文与元数据分表能避免 `raw_news_item` 承担过多职责。

### 候选与发布扩展

`candidate_article` 建议后续扩展：

| 字段 | 说明 |
| --- | --- |
| `content_preview` | Admin 可审核的正文片段预览 |
| `has_content_text` | 是否有关联正文内容 |
| `content_compliance_status` | `PENDING/APPROVED/REJECTED/LINK_ONLY` |
| `content_compliance_note` | Admin 合规审核备注 |

`news_article` 是否保存授权全文需要再单独评估。阶段 25 推荐先不改用户端文章表，只让候选详情可查看正文预览和合规状态，发布时默认仍发布摘要、要点和原文链接。

## 后端实现方案

### 服务拆分

建议新增：

```text
ContentFetchService
ContentFetchPolicyService
HtmlContentClient
HtmlContentExtractor
RawNewsContentRepository
```

职责：

1. `ContentFetchPolicyService`：判断 source 和 raw item 是否允许抓取正文。
2. `HtmlContentClient`：有限 HTTP 获取，负责超时、大小限制、Content-Type 校验和代理环境兼容。
3. `HtmlContentExtractor`：从 HTML fixture 或真实页面中提取正文候选，清理脚本、样式、导航和 HTML 标签。
4. `ContentFetchService`：编排策略校验、HTTP 获取、正文抽取、长度裁剪、哈希去重、状态写入和错误记录。

### 流程设计

```text
Admin 手动触发 RSS source
-> RSS Provider 采集元数据
-> raw_news_item 入库
-> candidate_article 生成
-> ContentFetchService 可被单独触发
   1. 读取 raw_news_item + news_ingestion_source
   2. 校验许可、最新窗口、URL 和 source enabled
   3. 创建 raw_news_content PENDING
   4. 请求 original_url
   5. 校验响应类型和大小
   6. 提取正文
   7. 根据策略保存 snippet 或 fulltext
   8. 更新 raw_news_content SUCCESS/FAILED/SKIPPED
   9. 候选详情展示正文预览和合规状态
```

阶段 25 可以先支持后端服务和 Admin 详情展示，触发入口可以先放在后续实现中选择：

1. 后端接口：`POST /api/admin/candidates/{id}/content/fetch`。
2. 采集完成后手动二次触发，不在 RSS run 同步链路中自动抓取。
3. 后续阶段 29 再纳入异步队列。

推荐先做 Admin 手动二次触发，避免采集按钮等待网页抓取。

### HTTP 与解析边界

HTTP 限制：

1. 只允许 `http` 和 `https`。
2. 单次超时建议 5 到 10 秒。
3. 响应体大小默认上限 1MB，超过则跳过。
4. 只接受 `text/html` 或明确的文本页面。
5. 不发送登录 Cookie、用户 Token 或绕过访问控制的 Header。
6. 遵守 source 限流；连续失败 source 可自动暂停内容抓取。

正文提取：

1. 移除 `script/style/nav/footer/header/aside` 等非正文节点。
2. 优先读取 `article`、`main`、OpenGraph 描述、常见正文容器。
3. 清理 HTML 标签、重复空白和过短段落。
4. `SNIPPET_ALLOWED` 默认保存 500 到 1000 字。
5. `FULLTEXT_ALLOWED` 默认保存全文但设置最大长度，例如 20,000 字。
6. 提取结果太短时标记 `FAILED` 或 `SKIPPED`，不伪造正文。

### 错误处理

正文抓取失败必须可观测，但不能影响采集主链路：

1. 许可不满足：`SKIPPED`，错误信息说明策略原因。
2. 网络失败：`FAILED`，记录 HTTP/timeout 信息。
3. 非 HTML 或超出大小：`SKIPPED`。
4. 提取正文为空：`FAILED` 或 `SKIPPED`。
5. 重复正文：可复用已有内容哈希，避免重复保存。

如果正文抓取失败，候选仍可继续按 RSS 摘要进入人工审核。

## API 设计

候选详情扩展：

```json
{
  "content": {
    "captureMode": "SNIPPET",
    "fetchStatus": "SUCCESS",
    "preview": "正文片段...",
    "licensePolicy": "SNIPPET_ALLOWED",
    "licenseNote": "来源授权说明",
    "fetchedAt": "2026-06-25T10:00:00",
    "errorMessage": null
  }
}
```

手动抓取接口建议：

```http
POST /api/admin/candidates/{id}/content/fetch
Authorization: Bearer <admin-token>
Content-Type: application/json

{
  "mode": "SNIPPET"
}
```

响应：

```json
{
  "code": "OK",
  "data": {
    "candidateId": 123,
    "rawNewsItemId": 456,
    "captureMode": "SNIPPET",
    "fetchStatus": "SUCCESS",
    "preview": "正文片段...",
    "errorMessage": null
  }
}
```

约束：

1. 候选不存在返回 404。
2. source 未授权返回 422。
3. source 已停用返回 422。
4. 抓取失败返回 502 或返回 `FAILED` 结果，需要实现时二选一；推荐返回 `200 + FAILED`，让 Admin 可以直接展示失败记录。
5. 不允许前端传任意 URL，只能使用候选关联的 `raw_news_item.original_url`。

## 前端影响

Admin 候选详情页：

1. 展示 source 许可策略和授权说明。
2. 展示正文抓取状态：未抓取、已跳过、成功、失败。
3. 对授权 source 展示“抓取正文片段”或“抓取授权全文”按钮。
4. 对未授权 source 展示不可抓取原因。
5. 成功后展示正文预览，不直接开放用户端发布全文。
6. 发布前必须保留合规确认入口。

采集监控页：

1. 阶段 25 可以暂不新增内容抓取统计。
2. 后续阶段可增加内容抓取成功数、失败数和跳过数。

Flutter 用户端：

1. 阶段 25 不改变用户端展示。
2. 后续若展示授权全文，必须有明确 UI 设计和合规确认，不默认展开全文。

## 测试与回归方案

后端自动化测试：

1. `SUMMARY_ONLY` source 调用内容抓取返回跳过，不保存正文。
2. `LINK_ONLY` 和 `UNKNOWN` source 不保存正文。
3. `SNIPPET_ALLOWED` source 可从本地 HTML fixture 提取正文片段。
4. `FULLTEXT_ALLOWED` source 可从本地 HTML fixture 保存授权全文。
5. 缺少 `license_note` 时拒绝抓取正文。
6. 过期 raw item 不进入正文抓取。
7. 非 HTML 响应、超大响应、空正文分别记录失败或跳过。
8. 重复正文通过 `content_text_hash` 去重。
9. 候选详情可返回正文预览和抓取状态。
10. 发布接口不会把未审核正文直接发布到用户端。

Admin 测试：

1. 候选详情展示正文许可策略和授权说明。
2. 未授权 source 不展示抓取按钮。
3. 授权 source 点击抓取后显示 loading、成功预览或失败原因。
4. mock client 支持内容抓取状态流转。

回归命令：

```powershell
cd backend
.\mvnw.cmd test

cd ..\admin
npm test -- --run
npm run lint
npm run build
```

真实网页 smoke：

1. 必须显式开启，例如 `PULSEBRIEF_CONTENT_FETCH_LIVE_TEST_ENABLED=true`。
2. 只允许公开、无需登录、授权明确的网页。
3. 单次最多抓取 1 到 3 条，避免对来源造成压力。
4. smoke 结果记录 source、candidate/raw item、抓取模式、正文长度和状态。
5. 不提交真实抓取输出、网页快照或本地数据库 dump。

CI 规则：

1. CI 只使用本地 HTML fixture。
2. CI 不请求真实网页。
3. CI 不依赖代理、真实 RSS、AI Key 或 PDF 存储。

## 风险与分阶段落地建议

风险：

1. 正文抓取版权风险高，必须严格依赖 source 许可和授权说明。
2. 网页结构差异大，抽取质量不稳定，不能把正文抓取作为发布硬依赖。
3. 网络失败和反爬限制常见，必须失败可见但不阻断 RSS 元数据采集。
4. 保存授权全文会增加数据敏感性，需要后续清理、审计和发布控制。
5. 如果抓取服务与 Provider 耦合，会让 Provider 重新变成难维护的大采集器。

分阶段建议：

1. 第一批：只做设计、数据模型、策略校验和 HTML fixture 测试。
2. 第二批：实现 `raw_news_content`、`ContentFetchService` 和 Admin 候选详情展示。
3. 第三批：增加 Admin 手动抓取按钮和本地真实网页 smoke。
4. 第四批：再把内容抓取纳入阶段 29 的调度/队列。

## 当前决策

1. 阶段 25 采用独立 `ContentFetchService`，不修改 RSS Provider 职责。
2. 推荐新增 `raw_news_content` 表，不把正文直接塞进 `raw_news_item`。
3. 仅 `SNIPPET_ALLOWED` 和 `FULLTEXT_ALLOWED` 且有授权说明的 source 可以抓取正文。
4. 正文抓取失败不影响 RSS 元数据入库、候选生成和人工审核发布。
5. 阶段 25 默认不改变 Flutter 用户端展示。
6. 自动化测试只使用本地 HTML fixture；真实网页抓取只允许本地显式 smoke。
