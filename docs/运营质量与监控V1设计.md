# 运营质量与监控 V1 设计

## 背景与现状

PulseBrief 已完成真实资讯采集基础链路、候选生成、Admin 候选审核发布和每日简报管理。当前后端已有 `news_ingestion_source`、`news_ingestion_job`、`raw_news_item`、`candidate_article` 和 `news_article` 等表，采集任务会记录拉取数、新增数、重复数和错误信息。

现阶段 Admin 的“采集任务”仍是占位页，运营无法直接查看采集失败原因、今日采集表现或来源配置状态。继续推进真实资讯采集前，需要先补齐最低可用的运营监控能力，避免采集失败后只能依赖数据库排查。

## 目标与非目标

目标：

1. 在 Admin 中展示采集任务列表，尤其是失败任务的错误信息。
2. 展示今日基础统计：今日采集数、候选数、发布数、失败数。
3. 支持采集源启停配置，便于外部 API 异常时快速回退。
4. 记录发布操作日志，支持运营追溯文章发布、简报发布和简报下线。
5. 预留异常数据检测接口边界。
6. 保持后端接口可被 OpenAPI 文档识别，前端 API 客户端可在 mock/API 模式下切换。

非目标：

1. 本阶段不实现复杂调度中心、重试队列或任务编排。
2. 本阶段不接入新的外部资讯源。
3. 本阶段不实现真实权限角色体系，仍沿用现有 Admin Token。
4. 本阶段不做实时推送或 WebSocket 监控。

## 影响范围

后端：

1. 新增 `AdminIngestionController`，路径统一放在 `/api/admin/ingestion/**`。
2. 新增 Admin 采集应用服务，负责读取任务日志、来源配置和今日统计。
3. 补充 repository 查询方法和实体 getter。

Admin 前端：

1. 将“采集任务”占位页替换为真实页面。
2. 补充 Admin API client、类型定义和 mock 数据。
3. 展示统计卡、任务列表、失败原因、来源配置摘要和来源启停开关。

文档与测试：

1. 更新下一阶段任务清单和真实资讯采集总清单。
2. 增加后端 MockMvc 测试、Admin API client 测试和页面渲染测试。

## 数据模型或权限模型

复用现有数据模型：

1. `news_ingestion_job`：采集任务日志。
2. `news_ingestion_source`：采集源配置。
3. `raw_news_item`：原始资讯池。
4. `candidate_article`：候选资讯池。
5. `news_article`：已发布文章。

新增数据模型：

1. `admin_operation_log`：Admin 操作日志，首批记录发布相关行为。

本阶段不新增权限表。所有 `/api/admin/**` 接口继续由 `AdminTokenFilter` 校验 `Authorization: Bearer <admin-token>`。

## 后端实现方案

### 任务列表

接口：

```http
GET /api/admin/ingestion/jobs?status=FAILED&page=1&pageSize=20
```

响应字段：

```json
{
  "items": [
    {
      "id": 1,
      "sourceCode": "fixture-global",
      "triggerType": "MANUAL",
      "status": "FAILED",
      "startedAt": "2026-06-10T08:00:00",
      "finishedAt": "2026-06-10T08:01:00",
      "fetchedCount": 0,
      "newCount": 0,
      "duplicateCount": 0,
      "candidateCount": 0,
      "errorMessage": "Provider timeout"
    }
  ],
  "page": 1,
  "pageSize": 20,
  "total": 1,
  "hasMore": false
}
```

排序：按 `startedAt desc, id desc`。

### 今日统计

接口：

```http
GET /api/admin/ingestion/metrics/today
```

响应字段：

```json
{
  "fetchedCount": 42,
  "candidateCount": 18,
  "publishedCount": 6,
  "failedCount": 1
}
```

统计口径：

1. `fetchedCount`：当天开始的采集任务 `fetched_count` 求和。
2. `candidateCount`：当天创建的候选资讯数量。
3. `publishedCount`：当天发布时间或创建时间落在当天的已发布文章数量。
4. `failedCount`：当天开始且状态为 `FAILED` 的采集任务数量。

### 采集源列表

接口：

```http
GET /api/admin/ingestion/sources
```

响应字段：

```json
[
  {
    "id": 1,
    "code": "fixture-global",
    "name": "Fixture Global",
    "providerType": "FIXTURE",
    "enabled": true,
    "contentAccessPolicy": "SUMMARY_ONLY",
    "maxAgeHours": 24,
    "allowPdfDownload": false,
    "allowFullText": false
  }
]
```

### 采集源启停

接口：

```http
PUT /api/admin/ingestion/sources/{id}/enabled
Content-Type: application/json

{
  "enabled": false
}
```

响应字段与采集源列表单项一致。

规则：

1. 仅允许修改 `news_ingestion_source.enabled`。
2. `enabled=true` 表示后续采集调度可以使用该来源；`enabled=false` 表示调度和手动采集入口应跳过该来源。
3. 若来源不存在，返回 404。
4. 本阶段不新增审批流；发布操作日志只记录发布相关行为，不记录采集源启停。

### 发布操作日志

表：`admin_operation_log`

核心字段：

| 字段 | 说明 |
| --- | --- |
| `operation_module` | 操作模块，首批使用 `PUBLISH` |
| `action_type` | 动作类型：`PUBLISH_ARTICLE`、`PUBLISH_DIGEST`、`OFFLINE_DIGEST` |
| `target_type` | 目标类型：`ARTICLE`、`DIGEST` |
| `target_id` | 目标业务 ID |
| `target_title` | 目标标题快照 |
| `operation_status` | 操作结果，首批写入 `SUCCESS` |
| `operator_name` | 操作人，V1 使用 `dev-admin` |
| `detail` | 操作摘要 |
| `created_at` | 操作时间 |

记录点：

1. 候选资讯发布为文章成功后，记录 `PUBLISH_ARTICLE`，`target_id` 为发布后的 `news_article.id`。
2. 每日简报发布成功后，记录 `PUBLISH_DIGEST`，`target_id` 为 `daily_digest.id`。
3. 每日简报下线成功后，记录 `OFFLINE_DIGEST`，`target_id` 为 `daily_digest.id`。

查询接口：

```http
GET /api/admin/operation-logs?module=PUBLISH&page=1&pageSize=20
```

响应字段：

```json
{
  "items": [
    {
      "id": 1,
      "module": "PUBLISH",
      "actionType": "PUBLISH_ARTICLE",
      "targetType": "ARTICLE",
      "targetId": 101,
      "targetTitle": "高盛：AI 基建投资仍将持续",
      "status": "SUCCESS",
      "operatorName": "dev-admin",
      "detail": "候选资讯发布为用户端文章",
      "createdAt": "2026-06-10T09:30:00"
    }
  ],
  "page": 1,
  "pageSize": 20,
  "total": 1,
  "hasMore": false
}
```

## 前端影响

新增 `IngestionMonitorPage`：

1. 顶部为页面标题和说明。
2. 四个统计卡：今日采集、候选生成、已发布、失败任务。
3. 任务列表支持查看全部或失败任务，列表展示来源、触发方式、状态、计数和错误信息。
4. 右侧详情或来源摘要展示采集源启用状态、许可策略和最新内容窗口。
5. 采集源卡片提供启用/停用按钮，点击后更新状态并刷新启用来源数量。
6. 页面展示最近发布操作日志，帮助运营追溯发布与下线动作。
7. mock 模式下使用固定样例数据；API 模式调用后端接口。

## 测试与回归方案

后端：

1. `GET /api/admin/ingestion/jobs` 需要 Admin Token。
2. 任务列表返回任务状态、计数字段和错误信息。
3. 今日统计返回采集数、候选数、发布数和失败数。
4. 采集源列表返回启用状态和许可字段。
5. 采集源启停接口可修改启用状态，并返回更新后的来源配置。
6. 候选文章发布、每日简报发布和每日简报下线后写入发布操作日志。
7. 操作日志查询接口可分页返回最近发布行为。

Admin：

1. API client 能正确映射任务、统计和采集源字段。
2. 页面能展示统计卡、失败任务错误信息和采集源状态。
3. 页面能切换采集源启停状态。
4. 页面能展示最近发布操作日志。
5. mock 模式可独立渲染。

回归：

1. 后端全量 `mvnw test`。
2. Admin `npm test -- --run`、`npm run lint`、`npm run build`。
3. 必要时用 Playwright 截图检查采集任务页布局。

## 风险与分阶段落地建议

风险：

1. 目前采集服务只在成功路径记录任务完成，失败路径还需要后续统一捕获并写入 `FAILED`。
2. 操作人身份 V1 仍使用固定 `dev-admin`，后续接真实后台账号后需要替换为登录用户。
3. 采集源启停会影响真实采集，前端需要明确展示启用/停用状态，后续调度入口必须读取该状态。

分阶段落地：

1. 第一批：任务列表、失败日志展示、今日统计、采集源只读展示。
2. 第二批：采集源启停配置。
3. 第三批：发布操作日志和异常数据检测。
4. 第四批：调度失败重试、人工重跑和更细粒度告警。
