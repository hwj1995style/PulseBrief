# 采集调度、队列与重试设计

## 目标

阶段 29 将现有 Admin 同步手动采集扩展为可持久化的定时调度和失败重试链路，同时保留手动触发用于运营补采和本地 smoke。真实调度默认关闭，CI 不访问真实 RSS、网页、PDF 或 AI 服务。

## 调度边界

- 每个采集源独立配置是否调度、间隔分钟数和下次执行时间。
- 调度间隔限制为 5 到 1440 分钟，并继续遵守来源启停、小时限流和最新内容窗口。
- `MANUAL` 任务失败立即向 Admin 返回错误，不自动重试。
- `SCHEDULED` 任务失败进入 `WAITING_RETRY`，最多尝试 3 次。
- 重试采用 1、2、4 分钟指数退避，上限预留为 30 分钟。

## 持久化状态

采集源新增 `schedule_enabled`、`schedule_interval_minutes` 和 `next_run_at`。任务新增 `attempt_count`、`max_attempts`、`next_retry_at` 和 `cancel_requested`。

任务状态包括：`RUNNING`、`WAITING_RETRY`、`SUCCESS`、`FAILED` 和 `CANCELLED`。等待重试任务可由 Admin 取消；执行中的取消请求作为后续阶段的协作式中断信号，不强杀正在进行的网络请求。

## Worker

`IngestionSchedulingWorker` 每 30 秒轮询到期重试和到期来源。功能只有在 `PULSEBRIEF_INGESTION_SCHEDULING_ENABLED=true` 时注册；默认配置和 CI 均保持关闭。

本阶段继续使用现有同步 Provider 编排。RSS、正文、PDF、AI 和分类拆成多级异步队列属于下一轮增强，本阶段先确保采集入口具备持久化状态、重试、取消和可观察性。

## Admin 能力

- 来源 API 可开启/关闭调度并设置间隔。
- 任务 API 返回尝试次数、最大次数、下次重试时间和取消标记。
- 等待重试任务提供取消 API。
- 监控页展示任务尝试次数、下次重试时间及来源调度状态。

## 测试与风险

- 自动化测试只使用 Fixture Provider 和本地 MySQL。
- 默认关闭 Scheduler，避免测试上下文产生后台任务竞争。
- 数据库索引覆盖到期来源和到期重试查询。
- 多实例抢占、分布式锁和独立消息队列留到生产部署阶段；上线多实例前必须补充。
