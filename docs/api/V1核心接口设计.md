# PulseBrief V1 核心接口设计

## 背景与现状

PulseBrief 已完成 Flutter 8 个高保真 mock 页面和 Spring Boot 3 后端骨架。当前移动端页面仍直接消费本地 mock 数据，后端只有 `/api/health` 和 Flyway 初始表结构。下一阶段需要建立 V1 核心 API，让 Flutter 后续可以从 mock 数据平滑切换到 Spring Boot + MySQL。

现有后端能力：

1. Spring Boot 3.5.x。
2. Java 17。
3. Maven Wrapper。
4. Spring Web、Validation、Spring Data JPA。
5. Flyway + MySQL 8。
6. 初始表：用户、分类、来源、文章、订阅、收藏、阅读历史、播放历史、每日简报、推送任务。

## 目标与非目标

目标：

1. 定义用户端 V1 所需 API 契约：登录、分类、首页资讯、分类资讯、详情、简报、订阅、收藏、播放历史。
2. 先支持 mock 登录和种子数据，不接真实短信、邮件、资讯采集或 AI 服务。
3. 后端 Controller / Service / Repository 分层清晰，便于后续接真实采集和 AI 摘要。
4. Flutter 新增 Repository 层后可按同一模型消费 mock 或 HTTP API。
5. API 不返回新闻全文，只返回标题、来源、摘要、AI 摘要、结构化要点、影响分析和原文链接。

非目标：

1. 不实现真实短信验证码、邮箱验证码或三方登录。
2. 不实现真实 JWT 安全体系的完整生产配置，首轮可使用开发态 token。
3. 不接入真实 RSS、GDELT、投行报告或 AI 摘要服务。
4. 不实现后台管理端 API。
5. 不实现支付、会员、PDF 下载、全文缓存。

## 影响范围

后端：

```text
backend/src/main/java/com/pulsebrief
├── common
├── auth
├── category
├── article
├── digest
├── subscription
├── favorite
└── playback
```

数据库：

1. 复用 `V1__init_schema.sql` 已有表。
2. 新增 `V2__seed_v1_content.sql`，写入分类、来源、文章和简报种子数据。
3. 如登录 token 需要持久化，后续再新增 session 表；首轮不新增。

Flutter：

1. 新增 `core/network` 和 `shared/repositories` 或按 feature 拆分 repository。
2. 保留现有 mock 文件，增加 API 数据模型映射。
3. 页面暂不直接发 HTTP 请求，统一通过 repository 获取数据。

## 数据模型与权限模型

### 数据模型

核心表映射：

| 领域 | 表 | 用途 |
| --- | --- | --- |
| 用户 | `app_user` | mock 登录用户、游客用户展示 |
| 分类 | `news_category` | 首页 Tab、分类页导航、订阅标签 |
| 来源 | `news_source` | 资讯来源与合规策略 |
| 文章 | `news_article` | 首页、分类、详情、相关资讯 |
| 简报 | `daily_digest` | 每日简报页和播放器 |
| 订阅 | `user_subscription` | 我的订阅和个性化首页 |
| 收藏 | `user_favorite` | 文章收藏状态 |
| 播放历史 | `user_play_history` | 播放器和我的页面统计 |
| 阅读历史 | `user_read_history` | 我的页面统计 |

文章结构化字段：

1. `summary`：短摘要。
2. `ai_summary`：详情页 AI 摘要。
3. `key_points`：JSON 字符串数组，详情页核心要点。
4. `impact_analysis`：可能影响。
5. `original_url`：阅读原文入口。

### 权限模型

首轮采用开发态轻量鉴权：

1. 游客模式可访问分类、首页、分类资讯、文章详情、简报。
2. mock 登录返回 `accessToken`，客户端后续请求放在 `Authorization: Bearer <token>`。
3. 订阅、收藏、阅读历史、播放历史需要 token。
4. token 验证首轮可通过固定前缀和用户 ID 解析完成，后续替换为 JWT。

默认用户：

```text
userId: 1
nickname: Wenjin
```

## API 通用约定

基础路径：

```text
/api
```

统一响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {},
  "traceId": "dev-trace"
}
```

错误码：

| code | HTTP | 含义 |
| --- | --- | --- |
| `OK` | 200 | 成功 |
| `VALIDATION_ERROR` | 400 | 请求参数错误 |
| `UNAUTHORIZED` | 401 | 未登录或 token 无效 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `CONFLICT` | 409 | 状态冲突 |
| `INTERNAL_ERROR` | 500 | 服务端错误 |

分页响应：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 100,
  "hasMore": true
}
```

时间格式：

```text
yyyy-MM-dd'T'HH:mm:ssXXX
```

## V1 接口清单

### Auth

`POST /api/auth/login`

用途：手机号或邮箱验证码 mock 登录。

请求：

```json
{
  "account": "wenjin@example.com",
  "verificationCode": "123456",
  "agreementAccepted": true
}
```

响应：

```json
{
  "accessToken": "dev-token-1",
  "tokenType": "Bearer",
  "expiresIn": 604800,
  "user": {
    "id": 1,
    "nickname": "Wenjin",
    "avatarUrl": "",
    "bio": "每天几分钟，掌握全球脉搏"
  }
}
```

`POST /api/auth/guest`

用途：游客模式进入。

响应只返回游客用户信息，不返回持久 token。

### Category

`GET /api/categories`

用途：首页分类 Tab、分类页导航、订阅页推荐标签。

响应：

```json
[
  {
    "id": 1,
    "code": "finance",

    "description": "关注全球市场、利率、汇率、商品与资本动态",
    "sortNo": 20,
    "enabled": true

]
```

用途：首页。

查询参数：

| 参数 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `categoryCode` | 否 | `all` | 分类编码 |
| `pageSize` | 否 | `20` | 返回条数 |

响应：

```json
{
  "todayDigest": {
    "title": "今日全球简报",
    "subtitle": "精选 10 条全球重点资讯",
    "digestId": 1
  },

    "title": "高盛：AI 基建投资仍将持续",
    "sourceName": "Goldman Sachs Research",
    "categoryCode": "investment_view"

  "articles": []
}
```

`GET /api/articles`

用途：分类页列表、相关资讯列表。

查询参数：

| 参数 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `categoryCode` | 否 | `all` | 分类编码 |
| `page` | 否 | `1` | 页码 |
| `pageSize` | 否 | `20` | 每页条数 |

响应为分页文章卡片。

  "hot": true,

用途：收藏文章，需要登录。

### User Profile

`GET /api/user/profile`

用途：我的页用户资料和统计概览，需要登录。

响应：

```json
{
  "id": 1,
  "nickname": "Wenjin",
  "avatarUrl": "",
  "bio": "每天几分钟，掌握全球脉搏",
  "subscriptionCount": 4,
  "favoriteCount": 1,
  "readCount": 0,
  "playCount": 5
}
```

说明：

1. `subscriptionCount` 来自 `user_subscription`。
2. `favoriteCount` 来自 `user_favorite`。
3. `playCount` 来自 `user_play_history`。
4. `readCount` 本轮暂返回 `0`，后续阅读历史模块补齐。

## 后端实现方案

### 分层

1. Controller：参数校验、响应包装、HTTP 状态码。
2. Service：业务规则、权限判断、DTO 组装。
3. Repository：JPA 查询。
4. Entity：表映射。
5. DTO：请求与响应对象。
6. Common：统一响应、异常、开发态用户上下文。

### 首轮实现顺序

1. `common`：`ApiResponse`、异常处理、用户上下文。
2. `category`：分类实体、Repository、Controller。
3. `article`：文章卡片、首页、列表、详情。
4. `digest`：今日简报、简报详情。
5. `auth`：mock 登录和游客模式。
6. `subscription`：查询和保存订阅。
7. `favorite`：收藏与取消收藏。
8. `playback`：播放历史记录。

## 前端影响

Flutter 后续新增：

```text
mobile/lib/core/network/
mobile/lib/shared/repositories/
mobile/lib/shared/dto/
```

建议模式：

1. 页面依赖 repository，不直接依赖 HTTP client。
2. repository 首轮支持 `MockPulseRepository` 与 `ApiPulseRepository` 两种实现。
3. DTO 与现有 UI model 做一层 mapper，避免后端字段变化直接污染 widget。
4. API 接入前保持现有 mock 页面可运行。

## 风险与分阶段落地建议

风险：

1. 当前 `key_points` 和 `impact_analysis` 是文本字段，首轮可用 JSON 字符串，后续如需强查询再拆表。
2. 开发态 token 不是生产安全方案，只能用于本地 V1 骨架。
3. 种子数据需要遵守合规边界，不写入新闻全文。
4. Flutter API 接入过早会打断 UI 稳定性，应先完成后端契约测试。

分阶段：

1. 先提交本设计文档和测试方案。
2. 再实现后端 read-only API：分类、首页、文章、简报。
3. 再实现 user-write API：登录、订阅、收藏、播放历史。
4. 最后实现 Flutter Repository 接入开关。
