# PulseBrief 用户中心资料接口设计

## 背景与现状

Flutter API 模式已完成 8 页面页面级验收，当前唯一明确保留的页面级边界是“我的”页仍读取本地 `mockUser`，用户昵称、简介和统计数字没有通过 Spring Boot API 获取。后端已有登录开发 token、订阅、收藏和播放历史数据表，可支撑 V1 用户中心资料的最小闭环。

## 目标与非目标

目标：

1. 新增 `GET /api/user/profile`，返回我的页所需用户资料与统计数字。
2. 统计订阅数、收藏数、播放历史数来自真实数据库表。
3. Flutter Repository 新增 `getUserProfile()`，默认 mock 模式仍返回本地 mock 用户。
4. MinePage 改为通过 Repository 读取用户资料。
5. API 模式下“我的”页不再直接依赖 `mockUser`。

非目标：

1. 不实现真实用户表、注册体系、头像上传或 JWT。
2. 不实现收藏列表、阅读历史列表、播放历史列表页面。
3. 不新增支付、会员或设置保存能力。
4. 不在本阶段实现阅读历史列表管理；阅读历史模块补齐后，`readCount` 从 `user_read_history` 读取。

## 影响范围

后端：

1. 新增 `user` 包下 Controller、Service、Response。
2. 扩展 `UserSubscriptionRepository`、`UserFavoriteRepository`、`UserPlayHistoryRepository` 的 count 查询。
3. 补充 Controller 和 Service 测试。

Flutter：

1. `PulseRepository` 增加 `getUserProfile()`。
2. `MockPulseRepository` 返回 `mockUser`。
3. `ApiPulseRepository` 调用 `/user/profile` 并映射为 `UserProfile`。
4. `MinePage` 改为 StatefulWidget，加载状态、错误状态和真实用户资料。

## API 设计

### GET /api/user/profile

请求头：

```http
Authorization: Bearer dev-token-1
```

响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "id": 1,
    "nickname": "Wenjin",
    "avatarUrl": "",
    "bio": "每天几分钟，掌握全球脉搏",
    "subscriptionCount": 4,
    "favoriteCount": 1,
    "readCount": 7,
    "playCount": 3
  }
}
```

鉴权：

1. 缺少 Authorization 返回 401。
2. V1 继续使用 `DevTokenSupport` 解析开发 token。

## 测试与回归方案

后端测试：

1. 未登录访问 `/api/user/profile` 返回 401。
2. 登录用户访问返回资料和统计字段。
3. Service 聚合订阅、收藏、阅读历史和播放历史 count。

Flutter 测试：

1. `MockPulseRepository.getUserProfile()` 返回 mock 用户。
2. `ApiPulseRepository.getUserProfile()` 正确映射 API 响应。
3. `MinePage` 默认 mock 模式仍能显示 `Wenjin` 和统计字段。
4. live API 测试补充用户资料读取。

手工回归：

1. Android 模拟器 API 模式进入“我的”页，用户信息和统计来自 API。
2. 默认 mock 模式仍不依赖后端。

## 风险与分阶段落地

风险：

1. 当前没有真实用户表，昵称和简介仍由开发 token 的 userId 约定生成。
2. 阅读历史尚无数据表，本阶段 `readCount=0` 可能与原 mock 数字有落差。
3. 收藏和播放统计会随联调操作变化，截图数字不是固定值。

分阶段：

1. 本轮完成资料接口与 MinePage 接入。
2. 下一轮可实现收藏列表、播放历史列表和阅读历史列表。
3. 后续接真实登录体系时，再把资料来源替换为用户表。
