# PulseBrief 完整技术路线落地设计

## 背景与现状

PulseBrief 的产品设计文档明确了首版技术路线：Flutter + Spring Boot + MySQL。当前仓库已经有一个 React/Vite PWA MVP 和 Node mock API，用于验证产品信息架构、页面流程和 GitHub 工程基线。现在项目目标调整为按完整技术路线推进正式 V1 工程。

本机环境检查结果：

1. 当前默认 Java 是 Java 8，不适合直接运行 Spring Boot 3。
2. 本机未安装或未配置 Flutter SDK。
3. `mvn` 不在 PATH 中。
4. 本机存在 IntelliJ 自带 Maven：`D:\App\idealC\IntelliJ IDEA Community Edition 2025.1\plugins\maven\lib\maven3\bin\mvn.cmd`，但当前 `JAVA_HOME` 配置不一致，直接运行失败。
5. 本机有 `winget` 和 `choco`，可以用于安装 JDK 17 或 Flutter 依赖。

结论：本项目可以在本机采用多版本 JDK。保留全局 Java 8 给旧项目使用，PulseBrief 后端通过项目脚本临时启用 JDK 17。

参考来源：

1. Flutter Windows 安装文档：https://docs.flutter.dev/get-started/install/windows/mobile
2. Spring Boot 系统要求：https://docs.spring.io/spring-boot/system-requirements.html
3. Maven Wrapper 文档：https://maven.apache.org/wrapper/

## 目标与非目标

目标：

1. 建立正式单仓库工程结构：Flutter 用户端、Spring Boot 后端、MySQL、React 后台管理端。
2. Spring Boot 后端使用 Java 17，不污染全局 Java 8。
3. 后端使用 Maven Wrapper，避免依赖本机全局 Maven。
4. MySQL 使用 Docker Compose 作为本地开发默认启动方式。
5. Flutter SDK 本机独立安装，不和其他工具混放。
6. 保留当前 React/Vite MVP 作为过渡参考，逐步迁移真实功能。
7. 先交付 V1 核心闭环，再逐步补真实采集、AI 摘要和推送。

非目标：

1. 不在第一阶段实现 App Store / Google Play 上架包。
2. 不在第一阶段接入真实 APNs / FCM 推送。
3. 不在第一阶段接入完整 AI 生产服务，只保留摘要服务接口。
4. 不在第一阶段实现复杂个性化推荐、全文搜索、付费会员和研报 PDF。
5. 不强制修改系统全局 `JAVA_HOME`。

## 数据模型与权限模型

### 数据模型

第一阶段落地以下核心表：

1. `app_user`
2. `news_category`
3. `news_source`
4. `news_article`
5. `user_subscription`
6. `user_favorite`
7. `user_read_history`
8. `user_play_history`
9. `daily_digest`
10. `push_task`

表结构以产品设计文档第 11 章为基础，增加以下工程约束：

1. 所有业务表统一使用 `created_at`、`updated_at`。
2. 文章、简报、资讯源状态字段使用枚举字符串。
3. 外部来源内容只存标题、摘要、原文链接和结构化解读，不存完整正文。
4. `news_article.article_hash` 用于去重，初期由 `title + original_url` 计算。

### 权限模型

用户端：

1. V1 支持游客模式和模拟验证码登录。
2. 登录后由后端签发 JWT。
3. 收藏、订阅、阅读历史、播放历史必须携带用户 token。

后台端：

1. 后台 API 使用管理员 JWT。
2. 第一阶段只实现单管理员账号配置。
3. 所有后台写操作预留操作日志接口。

## 后端实现方案

### 技术选型

1. Java 17
2. Spring Boot 3.x
3. Maven Wrapper
4. MySQL 8
5. Spring Web
6. Spring Data JPA 或 MyBatis Plus，第一阶段推荐 Spring Data JPA，减少样板代码。
7. Flyway 管理数据库迁移。
8. springdoc-openapi 输出 API 文档。

### 本地多 JDK 策略

保留系统默认 Java 8。

PulseBrief 使用脚本临时切换 Java 17：

```powershell
.\scripts\use-jdk17.ps1
```

脚本职责：

1. 设置当前 PowerShell 会话的 `JAVA_HOME`。
2. 将 `%JAVA_HOME%\bin` 放到当前会话 PATH 最前。
3. 输出 `java -version` 供确认。
4. 不修改系统级环境变量。

建议 JDK 17 安装路径：

```text
D:\Dev\jdk\jdk-17
```

### Maven 策略

不依赖全局 `mvn`，后端项目生成并提交 Maven Wrapper：

```text
backend/mvnw
backend/mvnw.cmd
backend/.mvn/wrapper/
```

本机 IntelliJ 自带 Maven 可作为临时工具，但不作为项目要求。后端标准命令是：

```powershell
cd backend
.\mvnw.cmd test
.\mvnw.cmd spring-boot:run
```

## 风险与分阶段落地建议

### 风险

1. 本机 Flutter 安装可能需要 Android Studio、Android SDK 和模拟器，耗时较长。
2. Spring Boot 3 需要 Java 17，本机当前 Java 8 不能直接运行。
3. Maven 当前只发现 IntelliJ 自带版本，且受 `JAVA_HOME` 影响。
4. Flutter、Spring Boot、MySQL、React Admin 一次性全量推进会增加联调复杂度。
5. 真实资讯采集和 AI 摘要涉及版权、稳定性和成本控制，需要分阶段接入。

### 分阶段建议

第一阶段：工程底座。

1. 安装或配置 JDK 17。
2. 安装 Flutter SDK。
3. 创建 `backend/` Spring Boot 工程。
4. 创建 `mobile/` Flutter 工程。
5. 创建 `deploy/docker-compose.yml` 启动 MySQL。
6. 创建 `scripts/use-jdk17.ps1` 和 `scripts/use-flutter.ps1`。

第二阶段：后端核心 API。

1. Flyway schema。
2. 分类、文章、简报种子数据。
3. 首页、分类、详情、简报 API。
4. 收藏、订阅、播放历史 API。

第三阶段：Flutter 核心体验。

1. 登录和兴趣选择。
2. 首页、分类、详情、简报、我的。
3. 客户端 TTS 和底部播放器。
4. 接入后端 API。

第四阶段：后台管理。

1. 数据看板。
2. 资讯源管理。
3. 文章管理。
4. 简报管理。
5. 推送任务管理。

第五阶段：内容链路。

1. RSS mock 采集。
2. RSS 真实采集。
3. GDELT 采集。
4. AI 摘要服务抽象。
5. 简报定时生成。

第六阶段：推送与上架准备。

1. 推送任务发送器。
2. APNs / FCM 接入。
3. 用户协议、隐私政策、内容来源说明。
4. 上架材料。

## 环境决策

采用以下决策：

1. 本机允许多 JDK。
2. PulseBrief 固定 Java 17。
3. 不修改全局 Java 8。
4. 后端固定使用 Maven Wrapper。
5. Flutter 本机独立安装。
6. MySQL 本地优先用 Docker Compose。
7. 当前 PWA MVP 保留为参考原型，正式移动端由 Flutter 承接。
