# PulseBrief 真实资讯采集与 Admin 审核发布整体任务清单

## 总体目标

将 PulseBrief 从“种子数据驱动的用户端 V1”推进到“真实资讯采集 + 后台审核 + 用户端发布”的可运营闭环。

目标链路：

```text
外部公开资讯源
→ 采集任务
→ 原始资讯池 raw_news_item
→ 去重与归类
→ 候选资讯 candidate_article
→ Admin 人工审核编辑
→ 发布为 news_article / daily_digest
→ Flutter 用户端展示与播报
```

## 总体原则

1. 合规优先：V1 默认只采集公开标题、摘要、来源、发布时间、原文链接、公开元数据；仅对授权明确且属于最新窗口的来源采集正文片段、授权全文或授权 PDF。
2. 人工审核优先：V1 不做自动发布，所有真实采集内容必须经 Admin 审核后进入用户端。
3. 可追溯优先：发布后的文章必须保留来源、原文链接、采集时间、审核人和发布时间。
4. 小步落地：先做一个低风险采集源和一个审核发布闭环，再扩展更多数据源。
5. API 可替换：采集器与业务发布模型隔离，后续可替换 NewsAPI、GDELT、Finnhub、RSS 或自研采集器。
6. 最新优先：V1 只采集最新内容，新闻默认 24 小时窗口，公开报告和授权 PDF 默认 72 小时窗口，不做历史批量回溯。

## 第一轮推荐执行顺序

1. 阶段 11：真实资讯采集 V1 设计。
2. 阶段 12：采集适配层设计与首个 Provider。
3. 阶段 13：采集入库与去重。
4. 阶段 14：候选资讯生成。
5. 阶段 15：Admin API 审核发布设计。
6. 阶段 16：React Admin 工程骨架。
7. 阶段 17：候选资讯审核 UI 基础联调。

第一轮暂不做：

1. 真实 AI 摘要服务。
2. 真实音频生成。
3. 自动发布。
4. 多管理员复杂权限。
5. 付费研报、登录后 PDF 或历史研报库采集。
6. 未授权全文新闻转载。
