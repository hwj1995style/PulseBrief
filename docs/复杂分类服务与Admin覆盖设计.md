# 复杂分类服务与 Admin 覆盖设计

## 目标

阶段 28 将候选生成阶段原有的单段关键词判断拆分为可追踪的分类决策。系统给出建议分类、置信度和命中规则，Admin 保留最终确认权；用户端只读取人工确认后发布文章的最终分类。

## 决策优先级

1. 命中明确业务关键词时，按规则优先级生成建议分类。
2. 未命中关键词时，使用采集源 `default_category_code`。
3. 采集源不存在或未配置默认分类时，回退到 `global` 并标记低置信度。
4. Admin 可覆盖建议分类，但必须填写覆盖原因。
5. 分类建议不会绕过候选审核或自动发布。

首批规则覆盖：投行公开观点、AI 动态、科技趋势、宏观政策、财经市场和产业观察。规则顺序避免同一内容同时命中时产生不稳定结果，例如投行观点优先于 AI 主题。

## 数据模型

`candidate_article` 新增：

- `suggested_category_code`：系统建议分类。
- `classification_confidence`：0 到 1 的置信度。
- `classification_rule`：命中的规则或回退来源。
- `category_override_reason`：最终分类偏离建议分类时的人工说明。

现有 `category_code` 继续作为最终分类。发布为 `news_article` 时只复制最终分类，不向用户端暴露未经审核的建议。

## Admin 交互

候选详情展示建议分类、置信度和规则。运营人员选择与建议不同的分类时，页面显示“人工覆盖原因”输入框；原因为空时前端阻止保存和发布，后端再次校验并返回 422。

## 失败与降级

- 分类服务未命中规则时回退到来源默认分类。
- 来源默认分类不可用时回退 `global`，置信度为 0.30，继续进入人工审核。
- 低置信度不会阻断候选生成，但必须保留 Admin 确认入口。
- 分类失败不影响原始资讯入库、正文/PDF 处理和 AI 摘要任务。

## 验证范围

- 分类规则测试覆盖投行、AI、科技、宏观和财经样例。
- 候选生成测试验证建议分类、置信度和规则被持久化。
- Admin Controller 测试验证人工覆盖原因必填并可回读。
- Admin 页面测试验证分类覆盖原因输入和保存流程。
- 完整 CI 门禁继续覆盖后端、Admin、Flutter 和 Compose 配置。
