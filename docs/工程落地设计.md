# PulseBrief 工程落地设计

## 背景与现状

当前仓库只有《每日全球热点资讯APP产品设计文档.md》，内容覆盖 Flutter APP、Spring Boot 后端、MySQL、资讯采集、AI 摘要、语音播报、后台管理和推送。由于本地环境暂未安装 Flutter、Maven、GitHub CLI，首个工程交付先落地为可运行的 Web/PWA MVP：用 React + Vite 模拟移动端 APP 体验，用本地 mock API 和样例数据跑通核心产品闭环。

该交付不是替代最终 Flutter + Spring Boot 方案，而是为产品验证、交互走查、后端接口联调和 GitHub 初始提交提供可运行基线。

## 目标与非目标

目标：

1. 提供可本地运行的 PulseBrief 移动端 PWA 原型。
2. 覆盖首版核心体验：登录、兴趣选择、首页热点、分类浏览、资讯详情、每日简报、收藏、语音播报入口、我的页面。
3. 提供后台运营看板雏形，覆盖资讯源、文章、简报和推送的管理入口。
4. 提供与产品文档一致的 mock API 数据结构，便于未来迁移到 Spring Boot。
5. 补充测试、运行说明、Git 初始化和 GitHub 上传准备。

非目标：

1. 不在本阶段实现真实 RSS/GDELT 采集。
2. 不在本阶段调用真实 AI 摘要服务。
3. 不在本阶段接入 APNs、Android 推送或服务端 TTS。
4. 不在本阶段实现 MySQL 持久化服务。
5. 不在本阶段生成可上架的 iOS/Android 安装包。

## 影响范围

新增工程目录：

1. `apps/web`：React + Vite 移动端 PWA 和后台演示界面。
2. `services/mock-api`：Node 本地 mock API 服务。
3. `docs`：工程落地设计、测试方案、后续迁移说明。

保留原产品文档作为上层产品规格，不修改其核心业务设定。

## 数据模型与权限模型

MVP 使用前端内存状态和 mock 数据模拟以下模型：

1. `Category`：分类编码、名称、说明、订阅状态。
2. `Article`：标题、来源、分类、发布时间、热度、摘要、AI 摘要、核心要点、影响分析、原文链接、收藏状态。
3. `Digest`：简报类型、标题、时间、摘要、播报文案、关联条目。
4. `UserProfile`：昵称、订阅分类、播放设置、推送设置。
5. `AdminMetric`：采集数、有效文章数、简报数、活跃用户、播放次数、收藏次数、推送指标。

权限模型：

1. 用户端使用本地“游客/模拟登录”状态。
2. 后台端使用前端视图隔离，不做真实鉴权。
3. 迁移到 Spring Boot 时，用户端接口走 token，后台接口走管理员权限。

## 后端实现方案

本阶段提供 `services/mock-api`：

1. 使用 Node 原生 `http` 模块，减少依赖。
2. 提供 `/api/categories`、`/api/articles/home`、`/api/articles`、`/api/articles/:id`、`/api/digests/today`、`/api/admin/overview` 等接口。
3. 响应结构尽量贴近产品文档中的 API 清单。
4. 支持 CORS，便于 Vite 前端本地调用。

后续 Spring Boot 迁移建议：

1. 以 mock API 响应结构作为 DTO 初稿。
2. 用产品文档第 11 章 SQL 作为 MySQL schema 初稿。
3. 先实现分类、文章、简报只读接口，再补用户收藏、订阅和播放历史写接口。
4. 资讯采集、AI 摘要和推送作为独立阶段接入。

## 前端影响

本阶段实现 `apps/web`：

1. 默认移动端宽度优先，桌面端居中展示手机式工作台。
2. 底部四 Tab：首页、分类、简报、我的。
3. 详情页使用内部状态进入和返回。
4. 底部迷你播放器保持跨页面可见。
5. 后台视图通过顶部入口切换，展示运营数据和管理列表。
6. 使用 CSS tokens 定义颜色、间距、圆角、字体和阴影，方便未来迁移 Flutter Design Tokens。

## 测试与回归方案

1. 单元测试：验证排序、筛选、订阅数量、收藏切换、播报队列等业务逻辑。
2. 组件烟测：验证首页、分类、简报、我的页面可渲染关键内容。
3. 构建验证：执行 TypeScript 编译和 Vite build。
4. 手工回归：登录/兴趣选择、Tab 切换、详情打开、收藏切换、播放按钮、后台入口。

## 风险与分阶段落地建议

风险：

1. Web/PWA 与 Flutter 不是同一技术栈，后续需要迁移 UI 实现。
2. mock 数据无法覆盖真实资讯采集的异常情况。
3. 真实版权合规、推送策略和 AI 摘要质量需要后端接入后再验证。
4. GitHub 上传依赖本地或连接器具备仓库权限。

分阶段建议：

1. 阶段一：完成当前 Web/PWA MVP 和 mock API，上传 GitHub 建立工程基线。
2. 阶段二：补 Spring Boot + MySQL，只读接口替换 mock API。
3. 阶段三：实现 RSS/GDELT 采集、去重、分类和摘要任务。
4. 阶段四：实现 Flutter APP，复用本阶段交互与数据结构。
5. 阶段五：接入推送、真实 TTS 设置、合规页面和上架材料。
