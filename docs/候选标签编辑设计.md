# PulseBrief 候选标签编辑设计

## 背景与现状

React Admin 候选审核页已经支持候选列表、详情、发布、拒绝，以及发布前编辑标题、摘要、分类和来源。当前剩余缺口是“候选标签编辑”：运营人员无法为候选资讯补充如 `AI 基建`、`利率债券`、`大宗商品` 这类用于分发、检索和后续统计的轻量标签。

后端现有 `candidate_article` 与 `news_article` 均没有标签字段，前端如果只做本地标签输入会在刷新或发布后丢失，因此本轮需要补齐可持久化的数据模型和 Admin API 映射。

## 目标与非目标

目标：

1. Admin 可在候选详情中编辑候选标签。
2. 标签随 `PUT /api/admin/candidates/{id}` 持久化到 `candidate_article`。
3. 候选发布为文章时，标签同步保留到 `news_article`。
4. Admin API 列表、详情、编辑响应均返回标签。
5. React Admin mock/API 双模式均支持标签编辑。
6. 补充后端和 Admin 前端测试。

非目标：

1. 不在 Flutter 用户端展示文章标签。
2. 不实现独立标签字典、自动补全、热门标签统计。
3. 不做批量标签编辑。
4. 不改变现有分类模型，标签只作为分类之外的补充维度。

## 影响范围

后端：

- 新增 Flyway migration，为 `candidate_article` 和 `news_article` 增加 `tag_names` 字段。
- 扩展 `CandidateArticle`、`NewsArticle`、Admin Candidate DTO 和 mapper。
- 扩展 `AdminCandidateUpdateRequest`，接收 `tagNames`。
- 发布候选时将候选标签同步到 `news_article.tag_names`。

React Admin：

- `AdminCandidate` 增加 `tagNames: string[]`。
- 候选详情编辑区增加标签输入。
- 保存候选内容时提交标签。
- 详情区以轻量 tag chip 展示已保存标签。

Flutter：

- 本轮不改 Flutter UI，不消费标签字段。

## 数据模型

### `candidate_article.tag_names`

类型：`VARCHAR(512)`

含义：候选资讯的运营标签，使用英文逗号分隔存储，API 层以字符串数组返回。

约束：

1. 单个标签 trim 后不能为空。
2. 单个标签最长 24 个字符。
3. 最多 8 个标签。
4. 重复标签按首次出现保留。

### `news_article.tag_names`

类型：`VARCHAR(512)`

含义：文章发布后保留的运营标签。V1 不返回给 Flutter 用户端，后续可用于 Admin 文章管理、搜索、推荐和统计。

## 后端实现方案

1. 新增 `V8__candidate_article_tags.sql`：
   - `ALTER TABLE candidate_article ADD COLUMN tag_names VARCHAR(512) COMMENT '候选运营标签，逗号分隔';`
   - `ALTER TABLE news_article ADD COLUMN tag_names VARCHAR(512) COMMENT '文章运营标签，逗号分隔';`
2. `CandidateArticle`：
   - 增加 `tagNames` 字段与 getter。
   - `updateDraft(...)` 增加 `tagNames` 参数。
3. `NewsArticle`：
   - 构造方法增加可选标签重载或扩展现有构造参数。
   - 增加 `getTagNames()`。
4. `AdminCandidateUpdateRequest`：
   - 增加 `List<String> tagNames`。
5. `AdminCandidateResponse`：
   - 增加 `List<String> tagNames`。
6. `AdminCandidateApplicationService`：
   - 标准化标签并写入候选。
   - 发布文章时将候选标签写入文章。

## 前端实现方案

1. `AdminCandidate` 和 `AdminCandidateUpdateInput` 增加 `tagNames: string[]`。
2. API client：
   - HTTP 映射后端 `tagNames`。
   - Mock 保存标签状态。
3. 候选详情编辑区：
   - 增加“候选标签”输入框，使用中文逗号、英文逗号、顿号、分号或换行作为分隔符，允许标签内部包含空格。
   - 输入值保存时转换为 `string[]`。
   - 保存后以 tag chip 展示。

## 测试与回归方案

后端：

1. `PUT /api/admin/candidates/{id}` 保存标签并返回 `tagNames`。
2. 发布候选后，`news_article.tag_names` 保留候选标签。
3. 标签空值、重复值和过长值按规则标准化或拒绝。

Admin：

1. API client 测试覆盖 `tagNames` 请求和响应映射。
2. 页面测试覆盖输入标签、保存、详情展示。
3. 运行 `npm test -- --run`、`npm run lint`、`npm run build`。

后端验证：

```powershell
cd backend
.\mvnw.cmd test
```

## 风险与分阶段建议

风险：

1. 逗号分隔字段不适合复杂标签分析，但足够支撑 V1 审核编辑。
2. 用户端暂不展示标签，后续若展示需要补充 Flutter 模型和 UI。
3. 标签字典缺失会导致运营命名不统一。

分阶段建议：

1. 本轮先落地可持久化编辑和发布保留。
2. 后续在 Admin 文章管理页增加标签筛选。
3. 再后续视运营需要抽象独立标签表和自动补全。
